# Design-vs-Build Reconciliation Note (QA Finding 5.3)

**Date:** 2026-06-04
**From:** Kezia (QA)
**For:** Izy (Database & SQL) and Paul (Documentation & Integration)
**Re:** Mismatches between the design docs (data dictionary / ERD) and the actual build
(`database/profriends_inc.sql`). These are **not bugs** — the build is the source of truth
and works. The action is to update the documentation to match it.

---

## Items to reconcile

1. **`buyer.Address` is missing from the dictionary and ERD.**
   The SQL has `Address varchar(100) NOT NULL` on the `buyer` table, but it does not appear
   in the data dictionary or the ERD. **Action:** add `Address` (`varchar(100)`, NOT NULL) to
   both. (The buyer registration / add / edit forms now collect it — see finding 1.2.)

2. **`beneficiary` primary key is composite in the build, single in the design.**
   Build: `PRIMARY KEY (BeneficiaryID, BenF_Name)`. Design says the PK is `BeneficiaryID`
   alone. The composite key is intentional — it lets one `BeneficiaryID` group several named
   beneficiaries (the seed data shows `BEN-004` listed three times). **Action:** update the
   dictionary/ERD to show the composite PK `(BeneficiaryID, BenF_Name)`.

3. **ERD shows a phantom `SpouseID` for `spouse`.**
   The ERD draws a `SpouseID` primary key on the Spouse entity, but neither the data
   dictionary nor the SQL has it — the real PK is `BuyerID` (one spouse per buyer).
   **Action:** remove `SpouseID` from the ERD; mark `BuyerID` as the Spouse PK (and FK to
   `buyer`).

4. **`Birthplace` length disagreement.**
   Build: `Birthplace varchar(100)`. Dictionary: `varchar(50)`. **Action:** reconcile to one
   value. Recommend matching the build (`varchar(100)`) since that is what is deployed.

5. **Civil-status value set disagreement.**
   The manual/form lists Single / Married / Separated / Widower (no Divorced); the data
   dictionary lists five values including Divorced: **S**=Single, **M**=Married, **D**=Divorced,
   **P**=Separated, **W**=Widowed. **Action:** adopt the data dictionary's five values
   everywhere. The UI now shows the full words while storing the single-letter code
   (see finding 5.1).

## Not in scope here (already documented as known limitations)

- `users`, `properties`, `payments` are frontend-only mock data — no DB tables by design.
- Loan status (Pending/Active/Approved) is not persisted (no column).
- "Create Account" password is ignored — buyers log in with BuyerID + last name.
